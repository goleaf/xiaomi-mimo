## Migration-To-Schema Comparison

The domain migrations repeatedly use `$table->uuid('...')->constrained()`. In the live SQLite schema this created columns and indexes but not foreign-key clauses. Laravel's foreign-key helper is `foreignUuid()` for UUID references; therefore migration intent and live enforcement diverge across workspaces, memberships, projects, todos, children, pivots, comments, reminders, attachments, activity logs, and preferences.

The framework-style migrations use integer morphs/foreign IDs for passkeys, notifications, and personal access tokens even though `users.id` is UUID text. SQLite's dynamic typing can mask this in happy-path tests, but it is not a coherent schema contract.

Fresh migration execution is expected to complete but reproduce these structural defects. Corrective migrations on populated SQLite must preflight orphan/cross-workspace rows and use SQLite-compatible table rebuilds; simply adding a foreign key in place is not safe.

## Storage Guidance

The main SQLite file and its `-wal`/`-shm` companions must reside on a local filesystem with SQLite-compatible locking. Network filesystems that do not preserve SQLite locking and shared-memory semantics are unsupported. Backups must not be written under the public web root.

No schema, migration, index, or configuration change was made in this audit phase.
