# Security Audit

## Threat Model

The primary tenant boundary is an authorized workspace. Attackers considered here are unauthenticated clients, authenticated users outside a workspace, lower-role workspace members, API tokens with excessive permissions, and users submitting IDs/files from another workspace or filesystem location.

## Control Inventory

| Control                     | Verified state                                                                                                                                                                         |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Web authentication and CSRF | Product writes are in the `web` middleware group and receive session authentication and CSRF protection. Settings routes intentionally vary between `auth` and `auth+verified`.        |
| API authentication          | Public login/register issue Sanctum personal access tokens; remaining API endpoints use `auth:sanctum`.                                                                                |
| Login rate limiting         | Fortify defines an email/IP login limiter and the authentication test suite covers web login throttling. API login has no explicit limiter.                                            |
| Authorization registration  | `AuthServiceProvider` maps six policies. Label, tag, import, export, and backup policies are absent.                                                                                   |
| Validation                  | Fifteen Form Requests exist, but many domain writes still validate inline and several domain requests return `true` from `authorize()`.                                                |
| Model assignment            | Domain models define `$fillable` arrays; this reduces broad mass-assignment exposure but does not establish cross-workspace relationship validity.                                     |
| Notification lookup         | Mark-read resolves IDs through the authenticated user's notifications relation, preventing direct global notification lookup in that controller.                                       |
| File storage                | Attachments use generated UUID storage names on the public disk; backups use a non-public storage directory. MIME/extension/content and backup inventory controls remain insufficient. |

## Required Permission Decisions

The baseline authorization document proposes owner/admin/member behavior. Phase 2 must encode and test that matrix for workspace settings/deletion, invitations, membership/role/ownership changes, projects, tasks/assignment/deletion, labels/tags, activity, import/export, and backup/restore. Frontend visibility is advisory only; policies and scoped queries are authoritative.

## Residual Risk

No security finding is fixed by this audit. Until phase 2, cross-workspace write paths and unrestricted tokens are the most urgent application risks. Until phase 9, backup/restore endpoints should be treated as unsafe for production use.
